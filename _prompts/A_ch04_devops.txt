## 3-1. 퀴즈 생성의 수단 (입력·도구·출력 규약)

**입력 PDF (절대 경로)**
- `C:\pdf\ch04c_DevOps (2).pdf`

**도구 사용 규칙**
- Read 툴로 PDF를 직접 읽어라. PDF가 크면 `pages` 파라미터(예: `pages: "1-20"`)로 나눠서 전부 훑어라. 절대 일부만 보고 추측하지 마라.
- 필요하면 Grep/Glob/Bash를 써서 기존 deck 스타일을 참고해도 된다. 참고 파일: `C:\aa\quiz\quiz-data\mobile-decks\software-engineering-exam-prep\day-4-devops.json` (이건 "쓰레기가 많은" 샘플이므로 **복제가 아니라 반면교사**로만 봐라 — 특히 선지가 "배포 파일 이름만 정한다" 식으로 허술한 점, 기본/심화 쌍이 없는 점을 개선해야 한다).
- 최종 결과는 Write 툴로 **오직 하나의 JSON 파일**을 아래 경로에 저장하라:
  - `C:\aa\quiz\quiz-data\mobile-decks\software-engineering-exam-prep-v2\day-4-devops.json`
- 설명 문장, 마크다운, 코드펜스, 로그 파일 같은 부수적 결과물을 절대 만들지 마라. 오직 위 JSON 하나.

**출력 JSON 스키마 (엄격)**
```json
{
  "title": "Day 4 DevOps와 운영 자동화",
  "questions": [
    {
      "id": "devops-01",
      "type": "short" | "long" | "answer",
      "questionText": "…",        // type이 "short" 또는 "answer"일 때
      "longText":      "…",       // type이 "long"일 때
      "imageUrl":      "…",       // 선택. SVG/이미지 있을 때만
      "options": [
        { "text": "정답 …", "isCorrect": true  },
        { "text": "오답 …", "isCorrect": false },
        { "text": "오답 …", "isCorrect": false },
        { "text": "오답 …", "isCorrect": false }
      ]
    }
  ]
}
```

**스키마 규칙**
- `id`는 `devops-01`, `devops-02`, … 순서.
- `type`:
  - 질문 본문이 80자 이하 & 이미지 없음 → `"short"`
  - 질문 본문이 길거나 시나리오/코드/긴 지문 제시 → `"long"`
  - 순수 용어 암기형(정의만 물음) → `"answer"` (하지만 남발 금지)
- `options`는 **정확히 4개**. 첫 번째가 반드시 정답이 되도록 할 필요는 없다 — 툴이 나중에 셔플한다. 하지만 `isCorrect: true`는 **오직 1개**여야 한다.
- 오답(distractors)은 **그럴듯해야** 한다. "배포 파일 이름만 정한다" 같은 허수아비 선지는 감점. 학습자가 고민하도록 실제 가까운 개념(예: 애자일 vs 워터폴, CI vs CD, Blue-Green vs Canary 같은)에서 뽑아라.
- 정답 옵션의 `text`는 **그 자체로 충분한 설명**이 되도록 풍부하게 써라. 단답 1~2단어가 아니라 2~4줄 규모의 완결 문장이 기본. 자가완결성을 반드시 지킨다.
- 한국어로 출력. 원문 PDF의 영어 용어는 필요시 괄호 병기("지속적 통합(Continuous Integration, CI)") 해 학습 가치 상승.

**분량**
- 최소 30문제, 최대 60문제. PDF에서 추출한 **핵심 개념 수 × 2**(기본/심화 쌍)를 목표로.
- 너무 비슷한 문제(같은 개념 3개 변형)는 제거. 중복은 품질 저하.

**검증 체크리스트 (저장 직전 스스로 확인)**
1. JSON이 파싱되는가? (`JSON.parse` 기준)
2. `options.filter(o => o.isCorrect).length === 1`이 모든 문제에서 성립?
3. `questions.length` ≥ 30?
4. id가 `devops-NN` 연속?
5. 같은 개념의 [기본]/[심화] 쌍이 실제로 존재?
6. `questionText` 또는 `longText` 둘 중 정확히 하나만 채워졌는가?
7. 정답 text가 2줄 이상의 자가완결 설명인가?

## 3-2. 기타 (품질·스타일 가이드)

1. **반드시 PDF를 처음부터 끝까지 훑어라.** 특정 슬라이드만 인용하고 끝내지 마라. DevOps 챕터는 CALMS, CI/CD, 배포 전략(Blue-Green, Canary, Rolling), IaC, 관측(모니터링·로깅·트레이싱), SRE, MTTR/MTTD, Shift-left, ChatOps, GitOps, 컨테이너·오케스트레이션 연계 등 **다층 주제**를 담고 있다. 빠짐없이 커버.

2. **"[기본]/[심화]" 태그는 `questionText` 접두어로 명시하지 마라.** 태그는 내부 원칙일 뿐. 문제 자체의 난이도 배분으로 구현하라. 기본형은 정의/핵심 명칭을 묻고, 심화형은 원리·비교·트레이드오프·실무 시나리오를 묻는다.

3. **선지 설계 원칙**
   - 정답 오답 모두 비슷한 길이·문체로 작성(길이 편향 방지).
   - 오답은 "진짜 그럴듯한 오개념" — 예: Blue-Green 설명을 물을 때 Canary의 특징을 정답처럼 섞거나, CI의 설명을 CD로 착각하게 만드는 식.
   - "위 보기 모두 맞다" / "없다" / "알 수 없다" 같은 저급 선지 금지.
   - 정답이 "A와 B 둘 다" 형태면 두 개념의 관계를 정확히 설명하는 선지로 구성하라.

4. **본문(questionText) 작성**
   - "슬라이드의 핵심은?", "이 단원에서 중요한 것은?" 같은 메타 질문 절대 금지.
   - "DevOps란 무엇인가?" 보다는 "개발·운영 사일로를 해소하고 짧은 피드백 루프를 달성하기 위해 CALMS 다섯 요소를 내재화하는 접근법을 뭐라 부르는가?" 식으로 **개념의 작동 방식**이 문제 안에 드러나게.
   - 시나리오형(긴 지문 → 정답 개념 선택)도 적극 활용. 예: "A팀이 주 단위 수동 배포에서 잦은 회귀 버그와 핫픽스에 시달린다. 이를 완화하는 DevOps 관행 중 '변경을 주 수십 회 자동 배포해 작은 단위의 리스크로 관리'하는 것은?" → CD.

5. **교차 참조**: 같은 PDF 안의 그림/다이어그램이 개념 이해에 필수라면, 개념 이름만 텍스트로 재구성한 문제로 변환(파일럿에서는 이미지 삽입 생략).

6. **금지 사항**
   - 특정 회사/강사 이름, 슬라이드 번호, 특정 페이지 레이아웃 언급 금지.
   - 영어 원문을 그대로 옮긴 정의 금지(번역·재구성 필수).
   - 여러 개념을 한 문제에 욱여넣기 금지. 1문제 1개념 원칙.

7. **최종 품질 자체 평가**: 완성 후 본인이 이 퀴즈만 보고 DevOps 단원을 공부할 수 있는가? 아니면 다시 PDF를 열어야 하는가? 후자라면 해설·정답 텍스트를 보강하라.

## 3-3. 퀴즈 생성 원칙 (고정, 반드시 준수)

단순히 슬라이드의 제목이나 파편화된 단어를 묻는 퀴즈(예: "슬라이드의 핵심은?" -> "DevOps")는 절대 생성하면 안 돼. 다음 원칙을 반드시 지켜:
1. **가치 있는 개념 필터링:** 강의 자료에서 불필요하게 여러 문장으로 풀어쓴 특징이나 단순 수식어는 무시해. 명확한 정의, 핵심 원리, 장단점, 아키텍처, 개념 간의 비교 등 '독립적으로 가치 있는 개념'만 추출해.
2. **자가 완결성 (Self-contained):** 나는 강의 자료를 다시 보지 않고 '오직 네가 만든 퀴즈'만으로 공부할 거야. 따라서 질문은 구체적이어야 하고, 정답(또는 해설)란에는 해당 개념에 대한 풍부하고 완벽한 설명이 포함되어야 해.
3. **난이도 배분 (기본~심화):** 추출한 핵심 개념 하나당 다음 두 가지 수준의 문제를 모두 만들어.
   - [기본 개념]: 해당 용어/개념에 대한 직접적인 정의를 묻는 문제.
   - [심화 적용]: 해당 개념의 작동 원리나, 다른 개념과의 차이점, 실무적 장단점을 묻는 문제.
